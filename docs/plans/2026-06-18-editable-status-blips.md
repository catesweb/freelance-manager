# Editable & aligned status blips — implementation plan

**Goal:** Make the status blip in the Projects and Invoices list pages directly editable (click → dropdown → immediate save) and fix its alignment.

**Architecture:** The read-only `StatusBadge` becomes the content of a flat `Button` whose `MenuFlyout` lists selectable statuses. Menu clicks route through the view's code-behind to a new `SetStatusCommand` on the list ViewModel, which persists via the repository and reloads the list. Invoice "Overdue" stays derived (excluded from the menu); reloading recomputes effective status.

**Tech Stack:** Avalonia 11 (XAML + code-behind), CommunityToolkit.Mvvm (`[RelayCommand]`), EF Core repositories, xUnit tests with in-memory fake repositories (matching `ClientsViewModelTests`).

## Global Constraints

- .NET / dotnet SDK: invoke the real SDK at `C:\Program Files\dotnet\dotnet.exe` (the bare `dotnet` on PATH is a broken x86 stub). Run all `dotnet` commands from the repo root.
- Selectable project statuses: `Lead`, `Active`, `Complete`, `Archived` (all of `ProjectStatus`).
- Selectable invoice statuses: `Draft`, `Sent`, `Paid` only. `Overdue` is derived by `OverduePolicy` and must never be offered or stored.
- Status changes persist immediately and show a notification via `INotificationService` (`Success` on save, `Error` on exception), mirroring the existing `Save`/`Delete` command pattern.
- VM unit tests use in-memory fakes (the `ClientsViewModelTests` style), not `TestDb`; repository behavior is already covered by the repository test classes.

        }
    }

## Notes for the implementer

- The blip click selects its row before the menu acts: the code-behind reads the row from the clicked `MenuItem`'s `DataContext` and sets `Selected` explicitly, so the correct entity is updated regardless of DataGrid selection timing. If, during manual verification, the menu's `DataContext` does not resolve to the row, fall back to relying on `vm.Selected` (the DataGrid selects the row on click) — the command already guards on `Selected`.
- `[RelayCommand]` on an `async Task SetStatus(TStatus)` generates `SetStatusCommand` of type `IAsyncRelayCommand<TStatus>`; `.Execute(object?)` is used from code-behind and `.ExecuteAsync(TStatus)` from tests.
- Repository `UpdateAsync` for both entities follows the existing load-modify-save pattern already used by the `Save` commands, so a status-only change round-trips safely (invoice line items are replaced wholesale with the same loaded items).
